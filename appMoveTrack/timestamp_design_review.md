# 发送时标设计确认稿

## 1. 背景

当前轨迹文件中包含源数据时标。程序读取文件后，会使用该时标完成排序、分桶、构建回放时间轴、插值等处理。

后续 UDP 发送时，发送报文中的时标不应继续使用文件中的旧时标，而应替换为发送瞬间的当前系统时间。

本文件仅用于方案审核，不涉及代码修改。

## 2. 已确认语义

### 2.1 文件源时标

文件中的原始时标只用于数据处理流程：

- 读取文件
- 数据排序
- 按对象分桶
- 构建轨迹回放时间轴
- 插值定位

文件源时标不应作为最终发送报文的时标直接透传。

### 2.2 发送时标

发送时标在每次实际 UDP 发送时生成。

同一发送帧内的所有轨迹对象共用同一个发送时标。

发送时标使用本机系统时间，不使用文件中的旧时标。

### 2.3 发送时标字段

当前先确认两个字段：

| 字段 | 含义 | 格式 |
|---|---|---|
| `globalMs` | 当前本机系统时间的 Unix epoch milliseconds | 整数毫秒 |
| `UTC` | 当前本机系统时间字符串 | 默认 `yyyy-MM-dd HH:mm:ss.zzz` |

注意：`UTC` 只是字段名称，内容使用本机时间，不使用 UTC 时区转换。

### 2.4 `UTC` 格式

默认格式：

```text
yyyy-MM-dd HH:mm:ss.zzz
```

示例：

```text
2026-05-01 12:30:45.123
```

后续可能存在分支格式，例如不带毫秒：

```text
yyyy-MM-dd HH:mm:ss
```

具体分支规则后续再设计。

### 2.5 时间同源要求

`globalMs` 和 `UTC` 必须由同一次系统时间采样生成。

也就是说，应先获取一次当前本机时间，然后同时生成：

- `globalMs`
- `UTC`

避免两个字段之间出现毫秒级不一致。

## 3. 当前代码中的设计漏洞

当前发送链路中，`globalMs` 同时承担了两个语义：

1. 回放采样时间：用于 `Track::interpolateAt(globalMs)`
2. 协议发送时标：在 `RoomManager::protocolConvert()` 中写入 `globalMs` 字段

这两个语义不应共用同一个变量。

回放采样时间应只用于确定当前轨迹位置；发送时标应来自发送瞬间的本机系统时间。

建议后续实现时拆分为至少三个概念：

```cpp
sampleMs     // 回放采样时间，用于 interpolateAt(sampleMs)
sendEpochMs  // 本机系统时间 Unix epoch milliseconds，写入 globalMs
sendUtcText  // 本机系统时间字符串，写入 UTC
```

## 4. 暂缓设计项

以下内容本轮暂不处理：

### 4.1 其他字段名兼容

暂不处理以下字段名是否等价于发送时标：

- `utc`
- `timestamp`
- `time`
- 其他自定义字段名

当前先只把前面确认的 `globalMs` 和 `UTC` 语义做正确。

### 4.2 旧文本发送格式

旧文本格式暂时不变。

当前旧格式类似：

```text
TRACK,<objId>,<objType>,<lat>,<lon>,<alt>,<fitType>
```

是否扩展旧格式中的时间字段，后续单独设计。

### 4.3 调试辅助功能

`seek`、倍速播放等属于调试辅助功能，后续可能删除，不作为核心业务设计前提。

回放区间循环时，发送时标仍按真实本机时间向前走。

## 5. 后续实现边界建议

后续若进入代码修改阶段，建议遵循以下边界：

1. 不再把回放采样时间 `sampleMs` 写入协议字段 `globalMs`。
2. 每次实际发送 UDP 帧时，只采样一次本机系统时间。
3. 同一帧内所有对象共用同一个 `globalMs` 和 `UTC`。
4. `globalMs` 使用 Unix epoch milliseconds。
5. `UTC` 字段名保持不变，但内容使用本机时间，默认格式为 `yyyy-MM-dd HH:mm:ss.zzz`。
6. 文件原始时标继续保留用于数据处理，但不作为发送时标透传。
